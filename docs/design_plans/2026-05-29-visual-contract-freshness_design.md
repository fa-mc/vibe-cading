## Objective

Add a machine-readable manifest of visual-contract SVGs plus a CI-enforced
check that regenerates each committed `.agents/plans/*_design_*.svg` from its
source class and byte-compares it against the committed file (with a coverage
gate against unregistered SVGs), so the `committed == regenerable` invariant
can no longer silently break.

## Architecture / Approach

### Approach chosen

Two new Modules, plus one CI step, plus a TODO close:

**Module A — `visual_contracts.toml` (the manifest).** A dedicated top-level
TOML file, parallel to `build.toml`, registering one row per committed design
SVG. Proposed schema (mirrors `build.toml`'s `[[build]]` / `[build.params]`
shape so it is instantly familiar):

```toml
# Each [[contract]] entry registers one tracked .agents/plans/*_design_*.svg
# visual contract.  tools/check_visual_contract_freshness.py regenerates each
# via tools/preview.py and byte-compares against the committed file.
#   "svg"    : path to the committed SVG, relative to repo root.
#   "model"  : full dotted import path to the source class.
#   "view"   : a named view from tools/preview.py NAMED_VIEWS.
#   "params" : constructor kwargs (REQUIRED where the class has no defaults,
#              e.g. LegoTechnicBeam.length_in_studs).

[[contract]]
svg   = ".agents/plans/2026-05-15-lego-technic-beam_design_iso_ne.svg"
model = "vibe_cading.lego.technic_beam.LegoTechnicBeam"
view  = "iso_ne"
[contract.params]
length_in_studs = 5

# … 8 more entries (top/front beam views carry the same params; the four
# gauge entries carry no [contract.params] table — defaults).
```

All **9** rows from the requirements table are registered at creation. Entries
whose class takes only defaults omit the `[contract.params]` table.

**Module B — `tools/check_visual_contract_freshness.py` (the check tool).** A
CLI that:

1. Parses `visual_contracts.toml` (stdlib `tomllib`).
2. **Coverage gate (FR3):** globs `.agents/plans/*_design_*.svg`, diffs the set
   against the manifest's `svg` fields. Any tracked SVG not in the manifest →
   fail. Any manifest `svg` pointing at a missing file → fail.
3. **Regenerate-and-compare (FR2, FR4):** for each entry, call
   `tools.preview.export_previews(model, tmp_dir, params, views=[view])`. That
   call writes `<ClassName>_<view>.svg` into a `tempfile.TemporaryDirectory`,
   applying `_fix_svg_viewport` + `_round_svg_coords` internally — the *exact*
   pipeline that produced the committed file. Byte-compare the regenerated file
   against the committed `svg` path (`filecmp` / raw `bytes` read). Mismatch →
   record + fail.
4. **`--update` mode (FR5):** same render loop, but instead of comparing,
   overwrite the committed `svg` path with the regenerated bytes; report which
   files changed; exit 0.
5. **Failure output (FR7):** for each drift, print the committed path and the
   one-line refresh command. Final summary line `X / Y contracts fresh,
   Z drifted` plus the coverage-gate result.

The tool reuses `tools.preview.export_previews` and (transitively)
`tools.model_loader` — it duplicates **neither** the loader, the param parser,
the viewport fix, **nor** `_round_svg_coords`. This is the load-bearing reuse
decision: if the check rounded coords itself, that copy could drift from
preview.py's and produce false drift reports.

**Filename indirection (FR4) — the one subtle bit.** `export_previews()` always
names output `<ClassName>_<view>.svg` (e.g. `LegoTechnicBeam_iso_ne.svg`,
`MThreeClearanceGauge_iso_ne.svg`). The committed names follow the design-slug
convention (`2026-05-23-calibration-helper-generic_design_m3_clearance_iso_ne.svg`).
The tool resolves this by rendering into a temp dir and comparing the
*single returned Path* (`export_previews` returns the written paths list)
against the manifest's `svg` field — never relying on filename equality.

**CI step (FR6).** A new `Visual contract freshness` step in the `check` job of
`.github/workflows/ci.yml`, placed **after** the `Topology check` step (itself
after `Build smoke`), mirroring the topology step's structure:

```yaml
      - name: Visual contract freshness
        # Regenerates every .agents/plans/*_design_*.svg via tools/preview.py
        # and byte-compares against the committed file; fails on drift or on a
        # tracked design SVG missing from visual_contracts.toml.  Runs after
        # Build smoke / Topology check so CadQuery is installed + validated.
        run: python3 tools/check_visual_contract_freshness.py
```

(`python3` in CI resolves to the 3.11 + cadquery interpreter set up by
`actions/setup-python` — same one `build.py` and the topology step use.)

#### Module-depth dual-lens evaluation (`vibe/INSTRUCTIONS.md` §Deep-Modules)

Required for the two new Modules. The deletion test: *would inlining each into
its callers lose anything?*

**Module A — `visual_contracts.toml`.**
- *Lens (a) maintainer-locality:* The check tool is the only internal consumer,
  so a hardcoded Python list inside the tool would technically work. On lens (a)
  alone the manifest looks borderline-deletable.
- *Lens (b) contributor-locality:* This is where it earns its keep decisively.
  The manifest is the **missing SVG→regeneration-command registry the project
  never had** — the exact knowledge gap the RCA identified (filenames don't
  encode class/params/view; the mapping was buried in prose across briefs). An
  external contributor adding a new model family with a visual contract needs
  *one obvious declarative place* to register it, parallel to the `build.toml`
  they already learned. A Python list buried in a tool file is discoverable only
  by reading the tool; a top-level TOML next to `build.toml` is
  cold-start-orientation surface. **Verdict: keep — earns its keep on lens (b)
  even though lens (a) is thin.** This is exactly the "cold-start-orientation
  surface" + "contributor-extension contract" false-positive the deletion test
  is documented to mis-flag.

**Module B — `tools/check_visual_contract_freshness.py`.**
- *Lens (a):* It is invoked by CI and by contributors locally; it is not a
  library others import. As a *check tool* it is a leaf, but it concentrates
  the regenerate-compare + coverage-gate logic that would otherwise be inlined
  as fragile bash in `ci.yml` (the topology step shows how much logic a CI step
  can accrete). A Python tool is testable in isolation; bash-in-YAML is not.
- *Lens (b):* A contributor reproducing a CI failure runs the **same tool**
  locally — single source of truth for "how is freshness defined". The
  `--update` mode is the contributor-facing refresh affordance. **Verdict:
  keep — observability + contributor-reproduction seam.** It mirrors the
  established `tools/check_topology.py` precedent (a check tool wired into CI,
  reproducible locally), so it adds no new architectural concept.

Neither module is a speculative abstraction: both close a concrete, proven gap.

### Visual contract (CAD tasks)

**Carve-out — no design preview SVG for this task.** Per the
*Visual Contract Deliverable* rule's scope carve-outs in
`vibe/INSTRUCTIONS.md` ("Optional for: refactors, internal API changes,
additive-only changes that don't alter visual outcome, instruction / config /
tooling tasks"), this task ships **tooling + CI + a manifest** — it creates no
new CAD model class and changes no visible geometry. There is therefore no
geometry to preview and no axis/hole-pattern convention to guard. This is the
correct application of the carve-out, not an omission. (Ironically, the
deliverable's entire purpose is to *enforce* visual contracts on the model
tasks that do require them.)

### Alternatives rejected

- **Manifest as a Python list inside the check tool** — rejected on lens (b):
  defeats the contributor-discoverable registry that is half the value (see
  dual-lens above). The RCA explicitly calls out the missing registry as the
  gap to close.
- **Extend `build.toml` with `[[visual_contract]]`** — rejected (Designer
  recommendation; see Q1). `build.toml` is the "models that ship as STEP"
  manifest, parsed by exactly two consumers — `build.py` (`tomllib.load` at
  `build.py:35`) and the `ci.yml` Topology step (inline `tomllib` loop) — each
  iterating every `[[build]]` entry as a shippable single-solid body. Visual
  contracts are a *design-artifact* concern (a different
  lifecycle, a different audience, keyed by SVG path not STEP output). Mixing
  two concerns into one file risks a contributor adding a `[[build]]` entry and
  wondering why it needs an SVG, or a parser for one consumer tripping over the
  other's rows. Separation keeps each manifest single-purpose.
- **Front-matter / parsed block inside each design brief** — rejected: requires
  the check tool to parse free-form Markdown across N brief files (fragile,
  no schema), and there is no single place to enumerate all contracts for the
  coverage gate. A contributor would have to know the front-matter convention
  exists. Strictly worse than a declarative top-level TOML.
- **mtime lint** — rejected per RCA / requirements D1 (git does not preserve
  mtimes; meaningless on a fresh CI checkout).
- **Tolerance / perceptual SVG compare** — rejected: byte-exact is proven
  viable today and is the strongest, simplest signal (Out of Scope in req).

## Data & Interface Contracts

Domain integrity gate is **NO**, so a full data-contract section is not
required. The one interface contract worth pinning (so the developer does not
re-derive it): the tool consumes `tools.preview.export_previews` —

```
export_previews(model_path: str,
                out_dir: Path,
                params: dict | None,
                views: list[str] | None) -> list[Path]
```

returning the list of written SVG `Path`s (one per requested view). The tool
calls it with a single-element `views=[view]` and uses the single returned
Path as the regenerated-bytes source. This is an **existing, stable** public
function — no signature change is in scope.

The `visual_contracts.toml` schema is the only new data shape; it is specified
in the Architecture section above (`[[contract]]` with `svg`, `model`, `view`,
optional `[contract.params]`).

## Out of Scope

*(Mirrors the requirements; no new exclusions surfaced.)*

- No tolerance / perceptual SVG compare — byte-exact only.
- No mtime lint.
- No model code / `build.toml` geometry change; no new model class; **no new
  design preview SVG for this task** (documented carve-out, above).
- No richer manifest registry (hashes, dimensions, multi-view bundling) beyond
  one row per SVG.
- No `engine-api.yml` change; no README CI badge.
- No retroactive `--update` sweep at landing (the 9 are already fresh).

## Known Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| OCCT/CadQuery version bump changes tessellation → every contract suddenly "drifts". | This is *correct* behavior — it forces a `--update` refresh in the same PR that bumps the dep. Document this in the tool's failure message ("if you just bumped cadquery, run `--update`"). Not mitigated away; it is the intended guard. |
| Manifest format flipped by TL co-sign (Q1) after the design is approved. | The check tool's parse layer is ~10 LOC and isolated; switching from a dedicated TOML to a `build.toml` extension changes only the parse function + the file it reads. Low rework cost; flagged as the one pending-co-sign item. |
| `export_previews` writes `<ClassName>_<view>.svg`; a future class rename silently breaks the manifest's `model` resolution. | The tool fails loud on `load_class` `ModuleNotFoundError`/`AttributeError` (propagated from `model_loader`) — a renamed/removed class is a hard failure naming the dotted path, not a silent skip. |
| Two design briefs could legitimately want the *same* class+view at two committed paths (collision). | The manifest keys on the committed `svg` path (unique by filesystem), not on class+view — multiple rows may share a class+view and render to different committed files. No collision. |
| Coverage-gate glob (`*_design_*.svg`) misses a future design SVG with a different naming convention. | All current design SVGs match (confirmed by glob). The Visual Contract rule *mandates* the `_design_*.svg` naming, so the glob tracks the rule. If the convention ever changes, the rule + glob change together. |
| `tempfile.TemporaryDirectory` cleanup vs. Windows file-lock quirks. | CI is `ubuntu-latest` (only supported CI target); local dev is the Debian container. No Windows path in scope. |

## Phase B Blocking Finding — clean-environment reproducibility (2026-06-01)

**Status: BLOCKING. Supersedes the "ready to ship" verdict above.** Surfaced by
running the check inside a *fresh git worktree* (no local `.env` /
`print_profiles_user.json`) — i.e. the environment CI actually runs in. All prior

**Root cause (RCA, empirically confirmed).** The 5 drifting contracts are the
