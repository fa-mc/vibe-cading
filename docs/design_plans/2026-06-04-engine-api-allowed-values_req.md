## Problem Statement
The platform exposes `engine_api.json` (a tracked repo-root artifact generated by `tools/gen_engine_api.py` and consumed by the platform's `query_engine_api` MCP tool) as the machine-readable surface a downstream code-generation agent reads to construct calls into `vibe_cading` model classes. Today that surface describes each constructor/classmethod parameter only as `{name, type, required, default, units}` (`tools/engine_api/extractor.py:116-128`). For a large and consequential class of string parameters — `fit`, `head_type`, `drive_type`, `type_`, and the screw/standoff/nut `size` families — the type is the unhelpful `str`, and the **set of legal values is a closed enum** that exists only as docstring prose plus runtime `if/elif` branches (e.g. `fit` resolves across `{"clearance","slip","tap","captive","press","interference"}` depending on family; `size` sets live in per-class, differently-named dicts — `METRIC_SIZES`, `IMPERIAL_SIZES`, `PLASTIC_SCREW_SIZES`, `SET_SCREW_SIZES`, `cls.DIMENSIONS`, `PH_SIZES`, `TORX_SIZES`). From the consumer's perspective, their code-gen agent must **infer** these closed sets from prose and signatures, which is unreliable: it guesses values like `"M4"` or `"countersunk"` that are not actually supported, and only discovers the error at runtime via a `ValueError`. The consumer is requesting a **machine-readable closed-set declaration** so their agent can constrain its output (and surface a picker) deterministically instead of inferring. This matters now because the maintainer has acknowledged the RFC and opted to build it, and the change is cleanly additive — it can ship without breaking any existing 1.0 consumer.

## User Story / Motivation
As the **maintainer of a downstream code-generation consumer** (issue #6), I need each closed-set string parameter in `engine_api.json` to carry a machine-readable list of its legal values (and, where useful, a one-line gloss per value) so that my code-gen agent can constrain its generated calls to the supported set deterministically — rather than inferring enums from prose and signatures and emitting calls that fail at runtime.

Secondarily, as the **vibe-cading maintainer**, I need this declaration to be derived from a single, drift-resistant source of truth and guarded by CI, so that the published `allowed_values` cannot silently diverge from what the model classes actually accept.

## Functional Requirements
<!-- Numbered, unambiguous, testable. -->

1. **R1 — `allowed_values` field (additive).** The `params[]` schema MUST gain an **optional** field `allowed_values`, typed `list | null`. Its position in the serialized object MUST be the slot immediately **after** `units` (stable, diffable ordering). Semantics: **absent or `null`** = no closed set is declared for this parameter (free-form / open value); a **non-empty list** declares the exact legal set. An **empty list (`[]`) is INVALID** and MUST be rejected by the validator (R8a) — a declared-but-empty enum is a producer bug, not "free-form" (use `null` for free-form).

2. **R2 — `value_doc` field (additive, render-only).** The `params[]` schema MUST gain an **optional** field `value_doc`, typed `{value: one-line-str} | null`. It is a render-only consumer hint (e.g. a picker tooltip) and carries no machine semantics beyond R8c. When present, **every key MUST be a member of that param's `allowed_values`** (`value_doc` keys ⊆ `allowed_values`); a `value_doc` key absent from `allowed_values` is INVALID. `value_doc` MUST NOT be present when `allowed_values` is absent/`null`.

3. **R3 — Extractor derives `allowed_values` from a single declared source.** The extractor (`tools/engine_api/extractor.py`) MUST parse a single, explicitly-chosen source of truth into the `allowed_values` list and populate `Param.allowed_values` / `Param.to_dict`. The exact extraction mechanism (a `Literal[...]` annotation read from the AST vs a structured docstring tag parsed by the extractor) is deferred to **OQ1** and MUST be decided at co-design / the human gate before implementation. Regardless of mechanism, the extractor MUST remain a **pure-stdlib AST walker** (no CadQuery / runtime import — R12) and MUST NOT attempt to read runtime `ToleranceProfile` attributes or runtime dict-key sets, which are not literal AST nodes at the parameter's definition site.

4. **R4 — `value_doc` extraction parity.** Where the chosen mechanism (OQ1) and the chosen `value_doc` policy (OQ2) call for populating glosses, the extractor MUST derive `value_doc` from the same declared, drift-resistant source as `allowed_values` (not from a second, independently-maintained table that can drift). If OQ2 defers `value_doc` population, the field is still declared in the schema (R2) but emitted as absent/`null` for all params; the extractor MUST still emit it `null` rather than omitting the key, only if that is the established convention for optional params — see R5.

5. **R5 — Emission convention follows existing `units` precedent.** `allowed_values` and `value_doc` MUST follow the project's existing optional-field emission convention so the byte-stable `gen --check` gate (R10) is satisfiable. The extractor currently always emits `units` (as a key with a `null` value when absent — `extractor.py:127`); the new fields MUST adopt the **same** always-present-key-or-conditional-key convention as decided at design, and the choice MUST be applied uniformly so the serialized form is deterministic. (Design decision, not left to the Developer to guess: pick one convention and state it in the design brief.)

6. **R6 — Schema version bump in lockstep.** `SCHEMA_VERSION` MUST be bumped **1.0 → 1.1** in the extractor (`tools/engine_api/extractor.py:64`) **and** the validator's pinned constant MUST move in lockstep (the validator imports + hard-pins `SCHEMA_VERSION` and fails on any mismatch — `tools/validate_engine_api.py:175-184`). Adding an optional `params[]` field is a **minor** (additive, non-breaking) bump per the project's versioning policy; the bump MUST NOT remove or rename any 1.0 field.

7. **R7 — Validator: `allowed_values` shape & type-consistency.** The validator (`tools/validate_engine_api.py`) MUST assert, for every `params[]` entry that carries a non-`null` `allowed_values`:
   - **(a)** it is a **non-empty** list (empty list → fail, per R1);
   - **(b)** every entry's runtime type is consistent with the param's declared `type` — for the 1.1 in-scope set this means **every entry is a JSON string** (the in-scope params are all `str`; R9). The validator MUST NOT silently accept a numeric or boolean entry under a `str`-typed param.

8. **R8 — Validator: `default` membership & `value_doc` key-subset.** The validator MUST additionally assert:
   - **(a)** if the param carries a `default` AND a non-`null` `allowed_values`, the **default value MUST be a member of `allowed_values`** — and the check MUST first **strip the embedded source-literal quoting** from `default` before membership testing (string defaults are emitted via `ast.unparse`, so a default appears as `"'slip'"` — quote-wrapped — not `"slip"`; a naive membership test would always fail). A `null`/`None` default is exempt from this check.
   - **(b)** when `value_doc` is present, **every `value_doc` key MUST be a member of `allowed_values`** (`value_doc` keys ⊆ `allowed_values`, per R2);
   - **(c)** `value_doc` MUST NOT be present when `allowed_values` is absent/`null` (per R2).

9. **R9 — Explicit coverage policy.** The set of string parameters that receive `allowed_values` in 1.1 MUST be **explicitly enumerated** in the design brief (in-scope vs left free-form), NOT auto-inferred from defaults. (A `str` param having a default does NOT imply it is a closed enum.) The exact breadth — the obviously-enum, consumer-facing set (`fit`, `head_type`, `drive_type`, `type_`, and the screw/standoff/nut `size` families) vs an exhaustive sweep of every `str` param — is deferred to **OQ3** and MUST be decided at co-design / the human gate. Whatever set is chosen, **every** param in that set MUST carry a populated `allowed_values` in the shipped 1.1 artifact (no half-covered in-scope set).

10. **R10 — Regenerate & commit the new artifact bytes in the same PR.** Because the new field changes the serialized output, the committed `engine_api.json` MUST be regenerated (`python3 tools/gen_engine_api.py`) and the new bytes committed **in the same PR**. The PR MUST leave `gen --check` (the byte-diff drift gate, `tools/gen_engine_api.py:109-125`) **green** — i.e. the committed bytes equal the freshly-regenerated bytes.

11. **R11 — At least one new extractor / wire-contract unit test.** The PR MUST add **≥ 1** unit test that exercises the new extraction + emission path directly (there are **none** today — `tests/` has no extractor/wire-contract test; `tests/test_imports.py` only checks class importability, and `tests/tools/` has only `test_calibrate.py`). The test MUST assert that a parameter known to be in-scope emits a non-empty `allowed_values` whose entries are exactly the declared set, and SHOULD assert the negative (a free-form `str` param emits `allowed_values` absent/`null`). This closes the "field detected but never wired" risk class — that the schema declares `allowed_values` but the extractor never actually populates it.

12. **R12 — Pure-stdlib, zero new dependencies.** The extractor MUST remain importable and runnable with the **Python standard library only** (it currently imports only `ast`, `sys`, `dataclasses`, `pathlib` — `extractor.py:55-60`). The change MUST NOT add any third-party pip dependency to the extractor, generator, or validator, and MUST NOT introduce a CadQuery / runtime import into the extractor.

13. **R13 — CI stays green end-to-end.** After the change, the `engine-api` CI workflow (`.github/workflows/engine-api.yml` — `gen --check` then `validate`) MUST pass, the new validator assertions (R7, R8) MUST pass against the regenerated artifact, and the existing test suite MUST remain green.

## Non-Functional Constraints
- **Additive & 1.0-consumer-safe.** A consumer written against schema 1.0 MUST continue to function unchanged: it ignores the two new optional keys and reads the same `{name, type, required, default, units}` it always did. No 1.0 field is removed, renamed, retyped, or reordered ahead of `units`.
- **Byte-deterministic serialization.** The serialized artifact MUST be deterministic across runs (stable key ordering, stable list ordering of `allowed_values`) so the `gen --check` byte-diff gate is stable and diffs are reviewable. `allowed_values` ordering MUST be defined and reproducible (e.g. source-declaration order), not set-iteration order.
- **Drift-resistance.** `allowed_values` MUST derive from a source the model class *also* uses (or that is enforced against it), so the published enum cannot silently diverge from the values the class actually accepts at runtime. (The OQ1 mechanism choice is precisely a choice about *how strong* this drift-resistance is.)
- **Pure stdlib / no new pip deps** (also stated as R12 because it is testable).
- **Contributor ergonomics.** Whatever source-of-truth mechanism is chosen (OQ1), adding `allowed_values` to a *new* enum-bearing string param later MUST be a small, local, discoverable edit at the param's own definition site — not an edit to a distant central registry that a contributor will forget. (This is a primary OSS-onboarding concern and a key axis of the OQ1 tradeoff.)

## Known Domain Constraints
<!-- Hard invariants from the codebase / wire contract the TL must not violate. -->
- The extractor is a **pure-stdlib AST walker** (`extractor.py:55-60`); it imports no CadQuery runtime and therefore **cannot** read runtime `ToleranceProfile` attributes, runtime dict contents, or any value that is not a literal AST node at the parameter's definition site. Any mechanism that requires resolving a runtime object is out of bounds.
- **No central enum/CHOICES table exists today, and zero `Literal[...]` annotations exist** anywhere in `vibe_cading/` or `parts/` (verified: 0 occurrences). The valid sets live in heterogeneous, per-class forms: docstring prose, runtime `if/elif` branches, and per-class differently-named dicts (`METRIC_SIZES`, `IMPERIAL_SIZES`, `PLASTIC_SCREW_SIZES`, `SET_SCREW_SIZES`, `cls.DIMENSIONS`, `PH_SIZES`, `TORX_SIZES`). Consequence: **populating `allowed_values` requires a one-time annotation/declaration pass over the in-scope param sites (~32 `str`-param sites observed for the candidate set)** plus net-new extractor parsing. This is the dominant cost; it is **not** captured by the RFC's ~½-day self-estimate (which covers only the extractor mechanism). Realistic effort: mechanism ~½ day; full feature ~1–2 days depending on OQ3 breadth.
- **String defaults are emitted quoted.** `ast.unparse(default)` wraps a string default in its source quotes, so a default surfaces as `"'slip'"` (a quote-wrapped string), not `"slip"`. The R8a "default ∈ allowed_values" check MUST strip this quoting before membership testing.
- **The validator hard-pins `SCHEMA_VERSION`** (`validate_engine_api.py:175-184`) and fails on any mismatch; the extractor's `SCHEMA_VERSION` (`extractor.py:64`) is the single source. They MUST move together — bumping one without the other red-fails CI.
- **The `gen --check` gate is a byte-diff** (`gen_engine_api.py:109-125`): the new field WILL trip it by design, which is exactly why R10 requires committing the regenerated bytes in the same PR.
- **Versioning policy:** adding a new optional `params[]` field is a minor (additive, non-breaking) bump → 1.0 → 1.1.

- **Refactoring the per-class `size` dicts into a central registry** — even if OQ1 favors annotations, consolidating `METRIC_SIZES` et al. into one table is a separate refactor and is out of scope unless co-design explicitly folds a minimal version of it in to serve as the annotation source.

## Open Questions
<!-- These three MUST be resolved at co-design / the human design gate. Each is framed with a recommended default + tradeoff; none is decided unilaterally here. -->

- [ ] **OQ1 — Extraction mechanism (source of truth for `allowed_values`).**
  - **Approach A — `Literal[...]` annotations read from the AST** (the RFC's pick). The enum set is declared as `fit: Literal["clearance", "slip", "tap"] = "slip"` at the param's own definition site; the extractor reads the `Literal` subscript from the annotation AST node (which it already unparses — `extractor.py:472`). *Pro:* single source of truth co-located with the param, type-checker-enforced (mypy/IDE flag an out-of-set call), maximally discoverable for an OSS contributor adding a new enum param (the declaration is *where they're already typing*), and drift-resistant because the annotation is the same object the signature exposes. *Con:* requires a **one-time annotation pass over the ~32 in-scope `str`-param sites**, and `Literal` sets must be kept in agreement with any runtime `if/elif` branch by hand (the type-checker enforces *call sites*, not the internal branch table).
  - **Approach B/C — a structured docstring tag the extractor parses** (e.g. an `@allowed fit: clearance | slip | tap` line). *Pro:* touches **no signatures** — zero annotation churn on the public API surface. *Con:* introduces a **parse grammar** the extractor must own and that contributors must learn; it is **not** type-checker-enforced, so it can silently drift from runtime behavior; and it duplicates information that arguably belongs in the type. Lower discoverability than an annotation.
  - **Recommended default: Approach A (`Literal[...]`).** The one-time annotation pass is the right *kind* of cost for an OSS-bound codebase — it buys type-checker enforcement, IDE auto-completion, and a single co-located source of truth, and it makes the next contributor's enum param a one-line local edit. The docstring-tag approaches optimize for "don't touch signatures" but pay for it in a hand-rolled grammar and a contract that can lie. *Tradeoff to weigh at the gate:* if the maintainer is unwilling to take the annotation pass now, B/C ships the mechanism faster but accepts drift risk and lower contributor ergonomics.

- [ ] **OQ2 — `value_doc` (R2) scope.**
  - **Option 1 — ship the field, populate selectively.** Declare `value_doc` in the schema now but author glosses only where they add real value (the prime case is **fit grades**, whose vocabulary is heterogeneous across families — `clearance`/`slip`/`tap`/`captive`/`press`/`interference` — and genuinely benefits from a one-line "what this fit means"). Leave self-explanatory values (`head_type: "flat"`, `size: "M3"`) gloss-free.
  - **Option 2 — author exhaustive glosses for every value now.** Maximally informative for the consumer's picker, but a large authoring burden, much of it low-value (glossing `"M3"` adds nothing), and more surface to keep in sync.
  - **Option 3 — defer `value_doc` entirely to a later bump.** Ship only `allowed_values` in 1.1; add `value_doc` in 1.2 when there is demonstrated consumer demand. Smallest 1.1, but the consumer gets bare values with no human-readable hint, and a second schema bump is needed later.
  - **Recommended default: Option 1 (ship the field, populate selectively — fit grades first).** It captures the high-value glosses (fit grades, where the value names are jargon) at low cost, declares the field so no future bump is needed to introduce it, and avoids the busy-work of glossing self-explanatory values. *Tradeoff:* a partially-populated `value_doc` means the consumer's picker shows tooltips for some values and not others — acceptable, since `value_doc` is explicitly a render-only hint (R2), not a guarantee.

- [ ] **OQ3 — Coverage breadth (R9): which `str` params get `allowed_values` in 1.1.**
  - **Option 1 — the obviously-enum, consumer-facing set.** `fit`, `head_type`, `drive_type`, `type_`, and the screw/standoff/nut `size` families (`from_size(size=...)` across `MetricMachineScrew`, `ImperialMachineScrew`, `PlasticsScrew`, `WoodScrew`, `SetScrew`, `HexStandoff`, `MetricHexNut`/`SquareNut`/`TNut`, `HeatSetInsert`, the drive `from_size` sites, etc.). This is the set the consumer's code-gen agent most needs constrained, and the set whose closed-ness is unambiguous.
  - **Option 2 — an exhaustive sweep of every `str` param.** Maximizes coverage but forces a judgment call on every `str` param about whether it is *truly* a closed set (R9 forbids auto-inferring enum-shape from a default), inflates the annotation pass well beyond ~32 sites, and risks mis-declaring a genuinely free-form `str` as a closed enum.
  - **Recommended default: Option 1 (the obviously-enum, consumer-facing set).** It delivers the consumer's actual ask (issue #6 names exactly these families), keeps the one-time annotation pass bounded and reviewable (~32 sites), and avoids the false-enum risk of sweeping every `str`. Any param outside the set stays free-form (`allowed_values` absent/`null`), which is honest. *Tradeoff:* a contributor wanting a *new* str param constrained must add the declaration themselves later — but per OQ1-Approach-A that is a one-line local edit, so the incremental cost is small. **Explicit guard (from R9): do NOT auto-infer enum membership from the presence of a default** — each in-scope param is chosen deliberately and listed in the design brief.
