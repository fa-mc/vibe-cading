# TL Architectural Assessment — RFC #41: `vibe_cading.mcp` subpackage

**Role:** TL (Technical Lead / Architect)
**Date:** 2026-06-17
**Status:** Advisory input for a maintainer-facing RFC recommendation comment. NOT a design brief, NOT an implementation.
**Engine SHA at RFC filing:** 8201564 (current `main` tip is 2d39985).

All code claims below are grounded in files I opened this session; file:line citations are inline.

---

## Bottom line up front

**Accept-with-modifications.** The *placement* and *import-isolation* parts of the RFC are architecturally sound and fit cleanly onto existing precedents (engine_api co-location, the established CI structural-guard pattern). The **dependency-policy claim is the one part that does not survive verification**: the RFC frames the `mcp` SDK as a lightweight, web-free "CLI-with-a-message-loop" dependency, but the official `mcp` PyPI package transitively drags in a full ASGI web stack (`starlette`, `uvicorn`, `sse-starlette`, `httpx`, `pydantic`, `pydantic-settings`, `python-multipart`) even when only stdio is used. That contradicts the RFC's own "closer to tools/ than to web code" justification and changes the cost calculus for making it mandatory at v1.

The two conditions that matter most before a design brief is written are in §6.

---

## 1. Placement — subpackage vs separate repo

**Verdict: co-location is the correct call. The argument is sound, and the engine_api precedent is a genuine (not strained) parallel.**

The RFC's core argument — "the tools are thin wrappers over engine class constructors/factories and must track their signatures; co-located, they evolve in the same commit as the classes" — is exactly the reasoning the codebase already committed to for `engine_api.json`. I verified the precedent is real and structurally identical:

- The engine_api artifact is generated by walking `vibe_cading/**` + `parts/**` with an AST extractor (`vibe_cading/tools/engine_api/extractor.py:185` `extract_classes`, driven by `vibe_cading/tools/gen_engine_api.py:60` `_build_payload`), and a CI `--check` gate fails on drift (`gen_engine_api.py` docstring lines 24-30). The whole point of that machinery is to keep a contract artifact byte-synchronized with the class signatures *in the same repo, same commit*. An MCP tool surface that wraps those same constructors has the identical skew problem and the identical solution.
- Concretely: the proposed `query_engine_class` / `list_engine_classes` tools are *reads of `engine_api.json`*. If they lived in a separate repo, they would consume a snapshot of `engine_api.json` and skew the moment a class signature changes here. Co-located, they ride the same `--check` freshness gate that already protects the JSON. This is the strongest single argument in the RFC and it holds.

**Coupling cost/benefit:**
- *Benefit:* zero cross-repo version-pinning dance; the MCP surface is testable against live classes in one CI run; "one repo, one story at the public debut" is a real onboarding benefit for a clone-only audience.
- *Cost:* the engine repo takes on a heavier dependency (see §2) and a new *runtime* concern (a server process) it did not have before. This is a real cost but it is *contained* by the import-isolation invariant (§3) — library consumers never pay it. The cost is paid only by someone who actually runs `python -m vibe_cading.mcp`.

**One nuance the RFC glosses:** the engine_api precedent is a *pure-stdlib, import-free AST artifact* (`extractor.py:18-19` — "imports nothing from CadQuery or the model packages themselves"). The MCP server is the opposite: it is a *runtime* that imports a heavy third-party SDK and (for the compile path) CadQuery itself. So "same reasoning as the extractor" is true for *why co-locate* (signature-tracking) but **not** for *what it costs* (the extractor costs nothing at import; the server costs a web-stack dependency tree). The maintainer should not let the clean extractor precedent launder the dependency weight. The fallback (separate `vibe-cading-mcp` repo) is correctly identified as worse on both coupling and clone-story; reject it unless the launch window genuinely can't absorb the work.

## 2. Dependency policy — `mcp` mandatory at v1 vs optional `[mcp]` extra

**Verdict: the RFC's reasoning is half-right, but its central factual premise is wrong. Recommend: `mcp` as an optional `[mcp]` extra *from day one*, NOT mandatory — OR, if mandatory, the maintainer must accept the web-stack tree with eyes open.**

### The factual problem

The RFC asserts (Proposal §3): *"the cadquery/OCP core dependency dwarfs the lightweight mcp SDK in install size, so 'keep installs lean' doesn't apply"* and (Transport §) *"Architecturally a CLI-with-a-message-loop, closer to existing tools/ entry points than to web code."*

I could not ground this by import — `mcp` is **not installed in this container** (`pip show mcp` → "Package(s) not found"; `importlib.metadata.version('mcp')` → `StopIteration`). So I verified against the upstream package.

**Confidence: high on the dependency list, medium on exact version numbers (web-sourced, not import-grounded).** Per the official `modelcontextprotocol/python-sdk` and its PyPI page, the `mcp` package's required dependencies are: `anyio`, `httpx`, `httpx-sse`, `jsonschema`, `pydantic`, `pydantic-settings`, `pyjwt`, `python-multipart`, `pywin32` (Windows), `sse-starlette`, `starlette`, `typing-extensions`, `typing-inspection`, `uvicorn`.

That is **a full ASGI web-server stack** (`starlette` + `uvicorn` + `sse-starlette` + `python-multipart` + `httpx`), pulled in *unconditionally* — the SDK does not gate the HTTP/SSE transport deps behind an extra, so a stdio-only consumer still installs uvicorn and starlette. This directly contradicts:
1. *"lightweight mcp SDK"* — it is not lightweight; it is a web framework plus a JSON-RPC layer.
2. *"closer to tools/ than to web code"* — the *vibe_cading* code may be stdio-only, but its dependency closure **is** web code (uvicorn is an ASGI HTTP server).
3. *"The stdio server is keyless by construction"* (Scope §) — true of the *vibe_cading* code, but `pyjwt` + the starlette auth surface ship anyway.

So the "cadquery dwarfs mcp" comparison, even if true on raw megabytes, is the wrong axis. The concern with a mandatory web stack is not disk size — it is **dependency-surface and supply-chain footprint** dragged into the import/resolve graph of a CAD library for *every* clone, including ones that will never run the server. CadQuery is load-bearing for the project's entire reason to exist; uvicorn/starlette are load-bearing for exactly one optional entry point.

### Stability of the dependency

`mcp` v1.x is a real, maintained, stable line (currently v1.27.x per web sources; a v2.0 is in alpha targeting ~2026-07). Taking a dependency on it is *reasonable* — it is the official SDK, not a fly-by-night package. **But pin conservatively** (`mcp>=1,<2`) because v2 is imminent and alpha; an unpinned `mcp` could pull a breaking v2 into a clone mid-launch. This pin decision is itself an argument for the extra: a breaking transitive bump in `pydantic`/`starlette` should never be able to break a `pip install vibe_cading` that only wanted the CAD library.

### Failure modes of each option

| Option | Failure mode |
|---|---|
| **Mandatory at v1** (RFC) | Every clone/`pip install` resolves uvicorn+starlette+pydantic-settings+pyjwt. A transitive break in that tree (e.g. a pydantic major bump, a starlette yank) breaks `import vibe_cading` for *library-only* users who never touch MCP. Larger attack surface in the dependency graph of a part-modeling tool. The import-isolation invariant (§3) protects the *runtime import graph* but NOT the *install/resolve graph* — the deps still resolve and install. |
| **Optional `[mcp]` extra from day one** | A clone-only user must know to `pip install -e ".[mcp]"` (or the repo must document `python -m vibe_cading.mcp` requires the extra). Mitigation is trivial: `python -m vibe_cading.mcp` can detect the missing import and print a one-line `pip install 'vibe_cading[mcp]'` hint (a 5-line `try/except ImportError` at the `__main__` entry). This is a *strictly better* failure mode than breaking library users. |

### On the "clone-only audience must run it out of the box" argument

This is the RFC's strongest pro-mandatory point and it deserves a real answer, not dismissal. For a **clone** (not a PyPI install), the user already runs `pip install -e .` against `requirements*`-style setup, so adding one documented extra (`pip install -e ".[mcp]"`) is a *one-line* delta in the clone instructions — not a barrier. The "out of the box" benefit is real but small; the cost (web stack in every library consumer's resolve graph) is borne by a larger population. The RFC's own deferral plan — "split mcp into an optional `[mcp]` extra later, at the PyPI moment" — concedes the extra is the right *end* state; doing it now costs ~10 lines of `pyproject.toml` + a docs line and avoids ever shipping the mandatory form. **There is no architectural benefit to the intermediate mandatory phase that the extra-from-day-one doesn't also deliver**, given clones run an editable install anyway.

`pyproject.toml` today has exactly one runtime dependency (`pyproject.toml:36-38`, `dependencies = ["cadquery"]`) and **no `[project.optional-dependencies]` table** — so adding the extra is greenfield, not a migration.

## 3. Import-isolation invariant — soundness AND enforceability

**Verdict: architecturally sound AND enforceable. This is the make-or-break, and it is well within the repo's existing CI-guard idiom. Recommend a two-layer guard (AST + live-import) modeled directly on the existing checks.**

The invariant ("`vibe_cading/mcp/` MUST NOT be imported by `vibe_cading/__init__.py` or any class module") is the right one and is already *half-satisfied by current design*: `vibe_cading/__init__.py` deliberately re-exports nothing from subpackages (`vibe_cading/__init__.py:20-24` — "intentionally does NOT re-export anything from sub-packages"). So the top-level package will not pull `mcp` in by default *today*. The invariant codifies and protects that property.

The repo already enforces three analogous structural invariants in `.github/workflows/ci.yml`, so the pattern is proven:
- **`check_no_main_blocks.py`** — an AST walker that forbids a forbidden construct in a tree with a carve-out for `vibe_cading/tools/` (`vibe_cading/tools/check_no_main_blocks.py:60` `find_violations`, exclude list at `:97`). This is *exactly* the shape an import-guard takes.
- A **grep belt-and-braces** twin for the same rule (`ci.yml`, "grep belt-and-braces" step).
- The **`ocp_vscode` import allowlist** (`ci.yml`, "Check no ocp_vscode imports outside view.py" step) — this is *already an import-restriction guard*: `grep -rlE '(from|import) ocp_vscode'` over `vibe_cading/ parts/` with a single-file allowlist. The MCP guard is the mirror image (forbid importing *out of* `mcp/`, restricted to *into* `__init__`/class modules).

### Concrete proposed guard: `vibe_cading/tools/check_mcp_import_isolation.py`

Two layers, because each catches a class the other misses:

**Layer A — static AST guard (primary).** Walk every `*.py` under `vibe_cading/` and `parts/` *except* `vibe_cading/mcp/` itself and `vibe_cading/tools/` (tools may legitimately reference mcp — e.g. a future `view`-like launcher). For each file, parse with `ast` and collect `ast.Import` / `ast.ImportFrom` nodes; fail if any names a module under `vibe_cading.mcp`. AST (not pure grep) so the rule does not false-positive on the string `"vibe_cading.mcp"` inside a docstring — the exact rationale `check_no_main_blocks.py:32` already cites ("AST (not regex) so string literals inside docstrings ... do not false-positive"). Add a grep belt-and-braces twin per the established pattern.

This catches the common case (a class module adds `from vibe_cading.mcp import ...`).

**Layer B — live-import assertion (defense-in-depth, catches the subtle case).** AST static analysis misses *transitive* / dynamic pollution — e.g. a class module imports a helper that imports `mcp`, or a `__init__.py` adds a re-export. A runtime check closes this:

```python
# pseudo: import the library surface in a clean subprocess, assert mcp absent
import subprocess, sys
code = "import vibe_cading; import sys; " \
       "leaked=[m for m in sys.modules if m.startswith('vibe_cading.mcp') or m=='mcp']; " \
       "sys.exit(1 if leaked else 0)"
subprocess.run([sys.executable, "-c", code], check=True)
```

Run it in a subprocess so `sys.modules` is pristine. This asserts the *actual property the invariant promises*: after `import vibe_cading` (and ideally after importing a representative class module), neither `mcp` nor any `vibe_cading.mcp.*` submodule is in `sys.modules`. This is the true "first-class ≠ tangled" proof — it tests the import graph, not just the source text.

**Wiring:** add both as a CI step in `.github/workflows/ci.yml` adjacent to the existing structural checks. Layer A is stdlib-only and runs in the lint job with no CadQuery needed; Layer B's library-surface import is cheap (the top-level `vibe_cading/__init__.py` only touches `importlib.metadata`, per `:35`). Note Layer B that imports a *class module* will pull CadQuery, so place that variant after the Build-smoke step where CadQuery is installed — or keep Layer B scoped to `import vibe_cading` (top-level) in the lint job and add the class-module variant post-build.

**This is genuinely make-or-break and it genuinely passes.** Without Layer B, "import isolation" is an aspiration; with it, it is a tested invariant. I would treat Layer B as a *blocking* requirement of the design brief, not a nice-to-have — it is the single check that converts the RFC's central promise into something CI can prove.

## 4. Tool surface & module structure — applying the Deep-Modules Dual-Lens Rule

**Verdict: the surface is right-sized and thin. Proposed module decomposition below. Every module earns its keep on at least one lens; I flag one that should NOT be created.**

### Reuse, do not reimplement

The non-negotiable architectural constraint: the MCP tools are **thin adapters over the existing loaders**, not reimplementations. The seams already exist and are clean:

- `query_engine_class` / `list_engine_classes` → **read `vibe_cading/engine_api.json`** (already deterministic, already CI-fresh). Do NOT re-walk the AST at request time; load the committed JSON. If "live" data is wanted, call `extract_classes` / `_build_payload` (`gen_engine_api.py:60`) — but the committed JSON is the contract and is cheaper.
- compile path → **`vibe_cading.tools.model_loader.instantiate(dotted, params)`** (`model_loader.py:186`) + `parse_params` (`:109`). This is *exactly* what the loader was built for — its docstring (`model_loader.py:72-79`) explicitly anticipates "a future caller that wants metadata only can use `load_class()` without `instantiate()`". An MCP compile tool is that caller. Reuse `load_solid` (`:233`) for the instance+solid composite.
- preview path → **`vibe_cading.tools.preview.export_previews(model_path, out_dir, params, views, quiet=True)`** (`preview.py:222`). The `quiet=True` flag (`:245-249`) was *already added for programmatic callers* (the freshness check) — the MCP server is the same kind of caller. Do not call `cq.exporters` directly.

If the MCP code reimplements param parsing, sys.path handling, or solid resolution, that is a review-blocking duplication finding under the project's "Generic Tooling / Utility Reuse" rule. The whole value of `model_loader.py` existing is that this server doesn't re-derive it.

### Proposed module layout

```
vibe_cading/mcp/
  __init__.py        # marker only; MUST NOT import server/tools (preserves isolation)
  __main__.py        # `python -m vibe_cading.mcp` entry; the ONLY place mcp SDK + the
                     #   try/except ImportError hint live. Wires stdio transport → server.
  server.py          # registers tools with the mcp SDK; maps tool calls → handlers.
  tools.py           # the thin handler functions (one per tool); pure adapters over
                     #   model_loader / preview / engine_api.json. NO mcp SDK import here
                     #   if feasible — keeps handlers unit-testable without the SDK.
  context.py         # get_design_context aggregation (see §5 — the one part with real
                     #   design weight).
```

### Dual-Lens evaluation of each proposed module

Per `vibe/INSTRUCTIONS.md` Deep-Modules Dual-Lens Rule (maintainer-locality AND contributor-locality):

- **`__main__.py`** — *earns it (both lenses).* Maintainer: it is the isolation firewall — the single file allowed to import the SDK, which makes Layer-A enforcement trivial (the guard simply doesn't exempt anything *except* this and `server.py`). Contributor: it is the cold-start orientation surface — "where does `python -m vibe_cading.mcp` start?" — a recognized keep-even-if-thin category in the rule.
- **`server.py` / `tools.py` split** — *earns it on contributor-locality.* Maintainer-locality alone is weak (a small server could inline handlers). But splitting the SDK-facing wiring (`server.py`) from the SDK-agnostic handlers (`tools.py`) means handlers are **unit-testable without standing up an MCP session** (see §5 testing), and an external contributor adding a new tool edits one pure function in `tools.py` against the documented loader API — they do not need to learn the mcp SDK's registration ceremony. This is the contributor-extension-contract category the rule says to keep even when current internal callers are few. **Keep the split.**
- **`context.py`** — *earns it (both lenses), and is the only module with non-trivial logic* (§5).
- **A `BaseMcpTool` ABC / per-tool class hierarchy** — *does NOT earn it. Do not create it.* The tool surface is 4 functions over stable loaders. A class hierarchy or `Protocol` for "an MCP tool" would be abstraction-for-its-own-sake: there is no polymorphic dispatch the SDK doesn't already provide, and an external contributor adding a tool benefits more from "write a function, register it in `server.py`" than from implementing an ABC. This fails *both* lenses — flag it as over-engineering if it appears in the design brief. (The deletion test: inlining a one-method ABC into 4 free functions loses nothing.)

### Compile/preview result transport over stdio

This is genuinely under-specified in the RFC and is a real design decision the brief must make. MCP tool results are JSON-RPC payloads (text-oriented). Options:

1. **File path return (recommended default).** The server runs *on the user's own machine* (RFC: "runs CadQuery on the user's own machine ... single-tenant, local trust"). So writing the STEP/SVG to a temp path and returning `{"path": "/tmp/.../Foo.step"}` is the natural, cheap, lossless option — the client is local and can open the file. This matches `export_previews` which already returns written `Path`s (`preview.py:253`, `:298`). **This is the right default given the local-trust model.**
2. **Inline base64** (STEP/STL binary) or **inline text** (SVG is XML text). Viable for small SVGs (the project's contracts are ~20–165 KB per `vibe/INSTRUCTIONS.md`), but base64-ing a STEP into a JSON-RPC frame is wasteful and bloats transcripts. Use only as an opt-in (`return_inline=true`) for clients without filesystem access.
3. MCP *resources* (the SDK's content-by-URI mechanism) — cleaner than base64 for binary, but adds SDK surface; defer to v2 of the tool contract.

**Recommendation:** default to file-path return (leverages local-trust + reuses `export_previews`' Path return), with an optional inline mode for SVG text. The brief must pin this — "how does the compile tool return its result" is exactly the kind of open question that, left to the developer, produces an inconsistent API.

## 5. Risks / blocking concerns / open questions for the maintainer

Ordered roughly by how blocking they are to writing a sound design brief.

**B1 — `get_design_context` has no single source and is the real maintenance-burden risk.** The RFC admits "no single source today — would aggregate `vibe_cading/lego/constants.py`, resolved `print_settings.get_profile()`, and `docs/` references." I verified all three are *independently-shaped, independently-versioned* surfaces:
- `constants.py` is a flat module of module-level floats (`vibe_cading/lego/constants.py:32+`, e.g. `STUD_PITCH`, `BEAM_THICKNESS`) — and it has *no machine-readable manifest*; an aggregator would either hardcode a curated subset (drifts) or reflect over the module (brittle, picks up private/derived names).
- `get_profile(name)` returns a `ToleranceProfile` dataclass (`print_settings.py:611`, `:265`) — already structured, the easiest of the three.
- `docs/` references are freeform Markdown — **there is no contract here at all**; "aggregate docs references" is hand-wavy and is where the maintenance burden hides. If `get_design_context` returns curated doc excerpts, every doc edit risks staling it with no freshness gate.

*This is the part of the RFC that most needs design before implementation.* Recommendation for the brief: define `get_design_context` as a **thin, explicitly-curated, versioned aggregate** with its *own* schema_version (like engine_api), sourcing (a) the `ToleranceProfile` via `get_profile()` live, (b) a *curated allowlist* of constants by name (not a reflection sweep), and (c) doc *pointers* (paths/anchors), NOT doc *content* — so it never has to track prose. Anything richer becomes a documentation-sync liability with no CI guard. **Flag to maintainer: do not let `get_design_context` become an un-gated second copy of the docs.**

**B2 — Tool-contract versioning is unspecified.** engine_api.json has `SCHEMA_VERSION` (`extractor.py:64`) with a documented additive-only bump discipline and a `--check` gate. The MCP *tool* contract (tool names, arg schemas, result shapes) is a *separate* public contract from engine_api.json and the RFC gives it no version field or evolution rule. An MCP client pins to tool signatures; renaming `query_engine_class`'s args silently breaks clients. The brief must state: tool-contract version, additive-vs-breaking policy, and whether tool schemas get a freshness/snapshot test.

**B3 — Error semantics over stdio are unspecified.** What does `compile` return when `instantiate()` raises (bad params, a CadQuery boolean failure, a missing class)? `model_loader` raises typed exceptions (`ModuleNotFoundError`/`AttributeError`/`ValueError` — `model_loader.py:159-182`) and `instantiate` lets constructor exceptions "propagate unchanged" (`:189-193`). The MCP layer must translate these into structured JSON-RPC errors (MCP `isError` / error content), NOT let a Python traceback escape as a transport-level crash. Define the error envelope in the brief: typed error code + human message + (optionally) which param was bad. This is a real surface a client depends on.

**B4 — Testing strategy for a stdio server.** A JSON-RPC-over-stdin/stdout loop is awkward to test end-to-end, which is *why* the §4 `server.py`/`tools.py` split matters: test the handler functions in `tools.py` directly (pure-Python, no SDK, no subprocess) for the bulk of coverage, and add *one* smoke test that spawns `python -m vibe_cading.mcp`, sends an `initialize` + one `tools/call` frame, and asserts a well-formed response. The brief must commit to this two-tier strategy; an MCP server with only end-to-end tests will be under-tested. Note the compile/preview handlers need CadQuery, so their tests run in the post-build CI stage (same placement logic as the existing topology/freshness steps in `ci.yml`).

**B5 — AGPLv3 implications of an MCP interface.** This needs a deliberate maintainer call, not a default. AGPL §13's network-use clause is triggered by "interacting with [the program] remotely through a computer network." The RFC's stdio transport is **local, single-tenant, no network listener** (Transport §) — so on its face §13 is *not* triggered (stdin/stdout is not "through a computer network"). That is good news and arguably an *argument for* keeping it stdio-only. **But** the moment someone wraps this in the deferred HTTP/SSE transport (which the `mcp` SDK's bundled uvicorn/starlette make trivially available — see §2), §13 *does* engage and the operator must offer Corresponding Source to remote users. The brief should (a) document that the stdio server is intentionally not a §13 network-interaction surface, and (b) flag that any future HTTP transport is an AGPL §13 event requiring a source-offer mechanism. Also: every new `.py` under `vibe_cading/mcp/` needs the AGPLv3 header per the project rule (non-empty files) — I confirmed the header shape on every file I opened (e.g. `model_loader.py:1-14`).

**B6 — `python -m vibe_cading.mcp` is the repo's first runnable subpackage.** The RFC notes this. Minor but real: it means `vibe_cading/mcp/__main__.py` is a new *kind* of file. The existing `check_no_main_blocks.py` forbids `if __name__ == "__main__":` guards under `vibe_cading/` *except* `vibe_cading/tools/` (`check_no_main_blocks.py:97`). A `__main__.py` does **not** use the `if __name__` guard (it runs on `-m` by construction), so it does **not** trip that checker — I verified the checker only matches the `Compare(Name('__name__'), Eq, Constant('__main__'))` AST shape (`check_no_main_blocks.py:43-57`). So no conflict. But the brief should confirm the new import-isolation guard (§3) carves out `vibe_cading/mcp/` itself (the server *is* allowed to import the SDK) — exactly as `check_no_main_blocks` carves out `tools/`.

**B7 — Dependency pin discipline (from §2).** Pin `mcp>=1,<2` (v2 is alpha/imminent). Whichever way the mandatory-vs-extra decision lands, the pin is mandatory.

## 6. Bottom-line recommendation

**Accept-with-modifications.** Placement: accept (co-location is correct; engine_api precedent is sound for the *why*, though not for the *cost*). Import-isolation: accept and *strengthen into a tested invariant*. Dependency policy: **modify** — the RFC's "lightweight, web-free" premise is factually wrong.

### The two conditions that matter most

**Condition 1 (dependency policy — the modification): make `mcp` an optional `[mcp]` extra from day one, not mandatory.** The RFC's justification for mandatory ("lightweight SDK, not web code") does not survive verification — the `mcp` package transitively requires a full ASGI stack (`uvicorn`, `starlette`, `sse-starlette`, `python-multipart`, `httpx`, `pydantic-settings`, `pyjwt`). For a clone-only audience that already runs an editable install, `pip install -e ".[mcp]"` is a one-line delta; that small cost is far better than dragging a web-server tree into the resolve graph of every library-only consumer. The RFC already concedes the extra is the right *end* state — there is no architectural benefit to the intermediate mandatory phase. Pin `mcp>=1,<2`. (If the maintainer overrides this and keeps it mandatory, that is a legitimate product call for the public debut — but it must be made *knowing* it ships uvicorn+starlette to everyone, not under the "lightweight" framing.)

**Condition 2 (import-isolation — the non-negotiable): the design brief must require the two-layer CI guard (`check_mcp_import_isolation.py`) — static AST + live-`sys.modules` subprocess assertion — as a blocking deliverable, modeled on the existing `check_no_main_blocks.py` / `ocp_vscode`-allowlist pattern.** This is the make-or-break for "first-class ≠ tangled." Layer B (the runtime `import vibe_cading` → assert no `mcp` in `sys.modules`) is what converts the RFC's central promise from an aspiration into something CI proves; it is cheap (top-level import only touches `importlib.metadata`) and must not be deferred.

Secondary (should be resolved in the brief, not blocking the accept): pin down `get_design_context` as a versioned, doc-*pointer*-not-doc-*content* aggregate (B1 — the real maintenance-burden risk); the tool-contract version field (B2); stdio error envelope (B3); the `server.py`/`tools.py` handler split for testability (B4/§4); compile-result transport = file-path-default given local trust (§4); AGPL §13 stdio-is-not-network note + future-HTTP flag (B5).

**Reject the fallback** (separate `vibe-cading-mcp` repo) unless the launch window genuinely cannot absorb the in-repo work — it is strictly worse on coupling and clone-story, as the RFC itself states.
