## Problem Statement
The gear generator under `vibe_cading/mechanical/gears/` ships `SpurGear`,
`HelicalGear`, and `RackGear`, but has no support for **bevel gears** —
conical gears that transmit motion between **intersecting** (non-parallel)
shafts. RC drivetrains and Lego-Technic adapters routinely need a 90° shaft
turn (differentials, steering takeoffs), which a parallel-axis spur/helical
gear cannot provide. A `BevelGear` class closes the obvious gap in the gear
family and lets the same parametric pipeline cover right-angle drives.

## User Story / Motivation
As a contributor building an RC/Lego drivetrain, I need a parametric
straight-bevel gear (and a matching meshing mate) so that I can transmit
torque around a 90° (or arbitrary) shaft angle without hand-modelling
conical involute teeth.

## Functional Requirements
1. The system MUST provide a `BevelGear` class in
   `vibe_cading/mechanical/gears/` that generates a straight-bevel involute
   gear as a single contiguous CadQuery solid exposed via a read-only
   `.solid` property (the `Gear` ABC contract).
2. `BevelGear` MUST inherit from the `Gear` ABC and reuse the shared 2D
   involute primitives (`Gear.gear_blank_with_teeth_2d`) rather than
   re-deriving involute math.
3. The constructor MUST accept `module`, `teeth`, `mate_teeth`, `face_width`,
   `bore` (composable `Bore` or legacy float), `pressure_angle`, and
   `shaft_angle` (degrees, **default 90.0**). The pitch-cone angle MUST be
   **derived** from `(teeth, mate_teeth, shaft_angle)`, not supplied raw:
   `δ = atan2(sin Σ, (mate_teeth/teeth) + cos Σ)`.
4. The tooth geometry MUST be generated by the **Tredgold scaled-section
   loft** method: a heel (large-end) toothed cross-section and a uniformly
   scaled toe (small-end) cross-section, lofted between two axis-perpendicular
   planes so all cones (root/pitch/tip) converge to the common pitch apex.
   The geometric simplification (common-apex / shared-apex cones) MUST be
   documented in the class docstring.
5. The zero-datum convention MUST place the **flat back / mounting face (heel,
   large end) at `Z = 0`**, with the body tapering toward the pitch apex in
   `+Z` and the gear axis on `+Z`, pitch circle centred on `(0, 0)`.
6. `BevelGear` MUST support a central through-bore along the axis via the
   existing composable `Bore` interface (`RoundBore`/`HexBore`/`DBore`/
   `KeyedBore`) and the legacy float-diameter form, cut cleanly through the
   full axial height with overcut (no zero-thickness wafer at either face).
7. `BevelGear` MUST override `mesh_with(other, phase)` to pose a meshing mate
   with **coincident pitch apexes** and the mate axis rotated by `shaft_angle`
   from self's axis, producing a visually-correct intersecting-axis pair for
   previews/assemblies (visual layout only, not a motion simulation — same
   disclaimer as the base method).
8. `from_iso(...)` MUST continue to work for `BevelGear` (module validated
   against `ISO_STANDARD_MODULES`, extra kwargs — `mate_teeth`, `shaft_angle`,
   `n_flank` — forwarded through `**kwargs`).
9. The class MUST expose a `demo()` classmethod returning a meshing bevel
   **pair** (the demonstration genuinely needs the two-instance mate pose,
   which a single-instance `view.py` invocation cannot show).
10. The class file MUST carry the AGPLv3 header, contain **no** `ocp_vscode`
    import and **no** `if __name__ == "__main__"` block (CI-enforced), and the
    class MUST be re-exported from `vibe_cading/mechanical/gears/__init__.py`
    alongside the existing gears.

## Non-Functional Constraints
- All dimensions in **mm**; no scaling of exported geometry.
- No new third-party pip dependencies (CadQuery + stdlib only).
- Derive every dimension from fundamental parameters / `lego.constants` —
  no magic numbers; document any overcut/clearance rationale inline.
- `python3 vibe_cading/tools/preview.py <…>.BevelGear` and
  `vibe_cading/tools/view.py … --demo` MUST run without error.
- Engine-API regeneration (`gen_engine_api.py`) MUST be re-run so the new
  class enters the wire contract deterministically.

## Known Domain Constraints
- Bevel-gear pitch angle (Σ=90° special case): `δ = atan(teeth/mate_teeth)`;
  a **miter** pair is `teeth == mate_teeth` at 90° → δ = 45°.
- Outer cone distance `A_o = pitch_radius / sin δ`; conventional face-width
  limit `face_width ≤ A_o / 3` — values beyond this MUST at least be validated
  against `face_width < A_o` (apex cannot be reached/overrun).
- Toe scale `s = (A_o − face_width) / A_o`; toe plane axial offset
  `= face_width · cos δ`; pitch apex on axis at `Z = pitch_radius / tan δ`.
- Tooth count undercut floor from `Gear.__init__` still applies to the real
  tooth count; document that the *virtual* (back-cone) tooth count
  `z_v = teeth / cos δ` is what governs true involute undercut, and note the
  approximation.
- Final solid MUST pass `assert len(result.solids().vals()) == 1`.
- Visual contract: a new model class with novel visible geometry (conical
  taper, axis convention) REQUIRES a co-located `_design_iso_ne.svg` (and a
  `_design_front.svg` is encouraged for the taper), registered in
  `visual_contracts.toml` and rendered label-free / `fdm_standard`.

## Out of Scope
- **Spiral-bevel** and **hypoid** gears (offset axes) — straight-bevel only.
- True **spherical-involute (octoid)** tooth profiles — Tredgold approximation
  is the accepted method for this task.
- Crowning, backlash-specific tooth thinning, and face/root-cone apex offset
  (the common-apex simplification is accepted).
- Physically-accurate rolling-mesh kinematics — `mesh_with` is visual only.
- A dedicated Lego-Technic bevel adapter / `build.toml` registration (a
  follow-up; `build.toml` is never auto-edited per project rule).
