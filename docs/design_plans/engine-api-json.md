# Design Brief: Structured `engine_api.json` (RFC from platform)

## Source

Platform-side RFC: [`fa-mc/vibe-cading` issue #2](https://github.com/fa-mc/vibe-cading/issues/2),
filed by `fa-mc/vibe-cading-platform`. RFC re-pinned to engine SHA `1c6ce1e`
after gate clarification.

## Objective

Emit a structured, machine-readable `engine_api.json` describing every public
model class in `models/**` so the platform's LLM code-gen agent can call
engine classes deterministically (constructor kind, params, types, units,
required/default, result accessor) instead of pattern-matching prose.

## Premise (resolved with platform on 2026-05-08)

`ENGINE_API.md` lives in the **platform repo**, generated by their own
docgen walking the engine submodule source. It has never shipped from
this repo — work is **additive**. Once `engine_api.json` ships upstream,
the platform deletes its docgen + `ENGINE_API.md` and re-points its MCP
`query_engine_api` tool at our artifact. Zero migration burden here.

## Readiness gates — all clear

- [x] Gate A — `ENGINE_API.md` provenance: platform-owned, additive.
- [x] Gate B — SHA reachability: re-pinned to `1c6ce1e`.
- [x] Gate C — FQN convention: bare `models.…`, no `engine.` prefix.

## 1. Tooling layout

Three files under `tools/` (all carry the AGPLv3 header per project rule):

- **`tools/engine_api/__init__.py`** — package init.
- **`tools/engine_api/extractor.py`** — shared library. Pure-`ast` walker;
  exposes `extract_classes(roots: list[Path]) -> list[ClassRecord]` and the
  `ClassRecord` / `Constructor` / `Param` dataclasses that mirror the schema.
  No CadQuery / no engine import — only stdlib.
- **`tools/gen_engine_api.py`** — CLI entry point. Calls
  `extractor.extract_classes([Path("models")])`, sorts deterministically
  (FQN ascending, params in source order), writes `engine_api.json` with
  `json.dumps(..., indent=2, sort_keys=False) + "\n"`. Flags: `--out PATH`
  (default `engine_api.json` at repo root), `--check` (regen to memory and
  diff against on-disk file; non-zero exit on drift — this is what CI runs).
- **`tools/validate_engine_api.py`** — schema validator. Loads
  `engine_api.json`, asserts schema invariants (see §5). Pure stdlib; no
  third-party `jsonschema` dependency. Returns non-zero on any failure.

Splitting the extractor as a library keeps the validator self-contained
(it doesn't need to re-extract) and gives future tooling a single import
point (e.g. an optional markdown renderer).

## 2. AST extraction rules

Walk `models/**/*.py` with `ast.parse`. For each top-level `ast.ClassDef`
that passes the discovery rule (§3), emit a record with the following
field derivations:

| Field | Derivation |
|---|---|
| `module` | Posix path of file relative to repo root, dotted, minus `.py`. |
| `name` | `ClassDef.name`. |
| `fqn` | `f"{module}.{name}"`. |
| `doc` | `ast.get_docstring(node)` verbatim, or `null`. |
| `constructors[]` | `__init__` (literal *or* synthesized from `@dataclass` — see below) plus every `@classmethod` whose name does not start with `_`. See sub-rules below. |
| `result_accessor` | See dedicated rule below. |

### `constructors[].kind`

- `kind = "init"` when `FunctionDef.name == "__init__"`.
- `kind = "init"` (synthesized) for `@dataclass`-decorated classes that
  have no literal `__init__`. The decorator generates `__init__` at
  class-creation time from the class-body `AnnAssign` fields; the
  extractor mirrors that statically. Rules: skip `ClassVar[...]`-typed
  fields, skip private-by-name (`_foo`) fields, treat fields with an RHS
  default as optional and ones without as required, emit the annotation
  via `ast.unparse` (verbatim, like literal init params). The synthetic
  record is prepended so "init first" ordering is preserved.
  In-tree consumers as of writing: `ToleranceProfile`,
  `HexVentilationGrille`, `SlottedVentilationGrille`, `RibbedKnob`,
  `ZipTieAnchor`, `PcbStandoffs`. The synthesized form has `doc = null`
  (the dataclass body has no function-level docstring to lift).
- `kind = "classmethod"` when the method has an `@classmethod` decorator
  (covers `from_size`, `b608`, `voron`, etc. — the dominant pattern in
  this repo).
- `kind = "factory"` is reserved for future module-level factory
  functions; the v1 extractor does not emit it. If we later add bare
  module-level helpers (e.g. `def make_bearing(...)`), they become
  `factory` records keyed by the module they live in. Document the
  reservation; do not implement.

### Same-module ancestor inheritance

When a discovered class defines neither an `__init__`, a public
`@classmethod`, nor a `@dataclass` decorator, the extractor walks **one
level up** the `bases` tuple and inherits constructors from a
same-module ancestor (looked up by bare-name match against the file's
top-level `ClassDef` index). This covers the
`MetricNylocNut(MetricHexNut)` pattern where the subclass only overrides
a class-level `DIMENSIONS` dict and relies on the parent's `__init__`.

Cross-file ancestor resolution is **out of scope for v1** — it would
require building an import graph, and no current case needs it. Bases
expressed as `ast.Attribute` (e.g. `pkg.Base`) are not resolved either.
A class that ends up with zero constructors after this lookup is
silently dropped from the artifact rather than emitted with an empty
`constructors` list (which would fail the validator's non-empty rule
in §5.4); the discovery rule (§3) is therefore implicitly tightened by
"must yield at least one resolvable constructor".

`constructors[].name` is the method name (`"__init__"` or e.g.
`"from_size"`). `constructors[].doc` is `ast.get_docstring(func)`.

### `params[]`

Iterate `func.args.args` (skip the leading `self` for `init` and `cls` for
`classmethod`). For each `ast.arg`:

- `name` — `arg.arg`.
- `type` — `ast.unparse(arg.annotation)` if present, else `"Any"`.
  Annotations stay verbatim (`"float"`, `"str"`, `"float | None"`,
  `"list[tuple]"`); the extractor does not normalise them.
- `default` — pair `func.args.args` with `func.args.defaults` (right-aligned,
  per Python semantics). If a default exists, emit `ast.unparse(default)`
  as a string (`"0.15"`, `"'PLA'"`, `"None"`). Otherwise omit the field.
- `required` — `true` iff no default present.
- `units` — see §9.1.
- `doc` — omitted in v1. Numpy/Google-style docstring parsing is
  out of scope for the first cut; the platform can fall back to the
  class-level docstring. Revisit if the platform asks for it.

### `result_accessor`

Default: `".solid"`. Override detection:

1. Walk class body for `FunctionDef` nodes decorated with `@property`.
2. If a property named `solid` exists → emit `".solid"`.
3. Else if exactly one public `@property` exists → emit `f".{name}"`
   (this catches `FastenerDrive.cutter` cleanly; see §9.3).
4. Else if **zero** public `@property`s exist → fall through to the
   default `".solid"`. The platform treats such records as opaque data
   carriers (e.g. `ToleranceProfile`, a `@dataclass` config object with
   no geometry surface). The validator does **not** flag this case; if
   the platform later wants stricter handling we can re-spin to emit
   `null` instead, but the literal-default reading is correct for v1
   and matches the shipped extractor.
5. Else if multiple public properties exist and none is `solid` → emit
   `null` and let the validator flag it as a maintenance issue.
6. Abstract base classes (any method decorated `@abstractmethod`) emit
   `result_accessor = null`; they have no instantiable result. The
   discovery rule (§3) excludes them anyway, but belt-and-braces.

### Module / class docstrings

Pass through verbatim from `ast.get_docstring`. No reflow, no markdown
parsing. The platform's renderer handles formatting.

## 3. Discovery rule

**Include any top-level `ClassDef` whose name does not start with `_` and
which is not abstract** (no `ABC` in bases, no `@abstractmethod` methods).
Rationale:

- The codebase has 68 public classes at module top-level and zero private
  ones (verified by `grep -c '^class _' models/`). Public-by-name is a
  stable proxy.
- An explicit allowlist would drift; requiring `.solid` would exclude
  `FastenerDrive` subclasses (`HexDrive`, `PhillipsDrive`, …) which the
  platform legitimately wants to call.
- `ABC` exclusion keeps `FastenerDrive` itself out (it is not directly
  instantiable) but its concrete subclasses in.

Nested classes are skipped (no current uses; would complicate FQN keying).
The validator asserts the discovery output is non-empty and contains no
duplicate FQNs.

## 4. Artifact location

**`engine_api.json` at the repo root, checked in.** Per the RFC's
"consumers pin SHA" stance, the artifact must be reachable at a stable
path for any commit on `main`. Root keeps it discoverable (siblings to
`README.md`, `build.toml`); `docs/` would imply human-readable; a release
artifact alone breaks SHA-pinning.

Add `engine_api.json` to the LFS-free, tracked tree. Do **not** add to
`.gitignore`. The CI gate (§6) ensures it stays in sync with source.

## 5. Schema validator scope

`tools/validate_engine_api.py` asserts:

1. File parses as JSON.
2. Top-level keys: `schema_version` (string, matches the validator's
   pinned expected value — bump in lockstep with §8) and `classes` (list).
3. Each class record has non-empty string `module`, `name`, `fqn`, and
   `fqn == f"{module}.{name}"`.
4. `constructors` is a non-empty list. Each entry has non-empty `kind` ∈
   `{"init", "classmethod", "factory"}` and `name`.
5. Every `params[]` entry has the keys `name` (str), `type` (str),
   `required` (bool). `default` is present iff `required is false`.
   `units` is a string or `null`.
6. `result_accessor` is a non-empty string starting with `.` **or**
   `null` (with a warning log line listing the FQNs that hit `null` so
   the maintainer can audit; warnings do not fail the run).
7. No duplicate FQNs.

Validator is invoked in CI and locally as a pre-flight before
`gen_engine_api.py --check`.

## 6. CI gate

**Current state:** `.github/workflows/` contains only `cla.yml`. There is
no test or lint pipeline yet — flag this back to PM. The engine-api gate
needs its own workflow file.

**Proposed `.github/workflows/engine-api.yml`:**

- Trigger: `pull_request` on paths `models/**`, `tools/engine_api/**`,
  `tools/gen_engine_api.py`, `tools/validate_engine_api.py`,
  `engine_api.json`. Plus `push` on `main` for the same paths.
- Steps: checkout → setup Python 3.x → `python3 tools/gen_engine_api.py
  --check` → `python3 tools/validate_engine_api.py`.
- `--check` regenerates in-memory and exits non-zero if the on-disk
  artifact differs (functionally equivalent to `git diff --exit-code
  engine_api.json` per the RFC, but doesn't require a git checkout
  step).

The wider question of "this repo has no test CI" is a separate task and
should be raised by PM as its own backlog item — not bundled here.

(b) when to bump `schema_version` (cross-ref §8), (c) what the CI gate
checks. Keep it under ~30 lines; defer deeper internals to inline
docstrings in `tools/engine_api/extractor.py`.

## 8. Versioning policy

`schema_version` follows semver-lite. **Major bump** when an existing
consumer would break: renaming a field (`params[].kind` →
`params[].sort`), removing a field, dropping a class previously listed,
making a previously optional param required, narrowing a `type`
annotation in a way that invalidates prior valid calls. **Minor bump**
when additive: introducing a new top-level field with a sensible default,
adding a new class to the `classes` array, adding a previously-omitted
optional param (e.g. when v2 starts emitting `params[].doc`), introducing
a new `constructors[].kind` value (consumers must already tolerate
unknown kinds).

The validator pins the exact expected `schema_version` string and
mismatches fail the build, forcing the version bump to be intentional.

## 9. Answers to the four RFC-deferred questions

### 9.1 Units convention

**Suffix- and bare-name-inferred mm/deg, with explicit override.**
Rule applied in extractor in this order:

1. If the parameter name is exactly `angle` or ends in `_angle` →
   `units = "deg"`.
2. If the parameter name is exactly one of the bare set
   `{bore, depth, diameter, height, length, pitch, radius, size,
   thickness, tolerance, width}` → `units = "mm"`.
3. If the parameter name ends in `_dia`, `_diameter`, `_radius`,
   `_length`, `_width`, `_height`, `_thickness`, `_depth`, `_offset`,
   `_clearance`, `_overlap`, `_tolerance`, `_pitch`, `_lift` → `units =
   "mm"`.
4. Else → `units = null` (consumer treats as dimensionless or
   self-evident — `count`, `size: str`, `material: str`, `mode: str`).

The bare-name carve-out was added after PR #3 review surfaced that 65 %
of params on `f261660` resolved to `null` because the suffix matcher
required a leading underscore. Bare `thickness` (e.g. `Bearing`) and
bare `length` (e.g. `MetricMachineScrew`) are common in this codebase.
`module` and `teeth` are deliberately excluded from the bare set —
`module` is the gear-module ratio (dimensionless), `teeth` a count.

This matches actual repo conventions (verified across `bearings.py`,
`standoffs.py`, `screws/metric.py`). An explicit `# units: <value>`
trailing comment on a parameter line could override the suffix rule, but
v1 ships without that — no current parameter requires it.

### 9.2 Required-vs-optional discipline

**No-default ⇒ required, no carve-outs.** The codebase is consistent:
every parameter that lacks a signature default is treated as a hard
requirement by the constructor body. There is no convention of "documents
optionality outside the signature". If a future case arises (e.g. a
`**kwargs` slot with semantic optionality), it should be explicit in the
signature, not in prose.

### 9.3 `result_accessor` uniformity

**Almost uniformly `.solid`. One documented outlier.**
`grep -rn "def solid\|def cutter" models/ --include="*.py"` yields:

- 39 classes expose a `.solid` property.
- `models/mechanical/screws/drives.py` exposes `.cutter` on
  `FastenerDrive` and its subclasses (`HexDrive`, `SlottedDrive`,
  `PhillipsDrive`, `TorxDrive`). These are subtractive-only by design;
  there is no positive `.solid` to expose.
- Various classes additionally expose `.to_cutter(...)` *methods*
  (parameterised cutters, distinct from a read-only accessor) —
  `Bearing.outer_pocket`, `MetricMachineScrew.to_cutter`, etc. These are
  methods, not result accessors, and are out of scope for this field.

The §2 detection rule (default `.solid`; fall back to the unique public
`@property` name; null when ambiguous) handles the `FastenerDrive`
family correctly without an allowlist.

### 9.4 Maintenance-doc home

Resolved in §7: **`CONTRIBUTING.md` section**, forced by `.agents/`
being git-ignored.

## 10. Phasing

**Single Developer cycle.** Total surface ≈ 250 LOC across three files
plus one workflow YAML and one `CONTRIBUTING.md` section. No
inter-dependent phases — extractor, validator, and CI gate land together
or the artifact is unenforced. Splitting risks shipping a generator
without a guard, which is the exact failure mode that lets the artifact
drift.

Suggested order within the cycle (single PR):

1. `tools/engine_api/extractor.py` + dataclasses.
2. `tools/gen_engine_api.py` (calls extractor; emits artifact).
3. Initial `engine_api.json` committed at repo root.
4. `tools/validate_engine_api.py`.
5. `.github/workflows/engine-api.yml`.
6. `CONTRIBUTING.md` section.

## Non-goals (per RFC)

- No runtime introspection.
- No constraint DSL / valid-range expressions.
- Not OpenAPI/JSON-Schema-strict.
- No human-readable markdown output (deferred; trivial to add later as a
  separate `tools/render_engine_api_md.py` consuming the JSON).

## Consumer commitments (from RFC, for reference)

- Platform pins `schema_version` and tracks bumps.
- Platform files concrete bug reports against `engine_api.json`.
- Platform does not depend on undocumented schema fields.
