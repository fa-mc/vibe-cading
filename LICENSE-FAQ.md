## Why AGPLv3 instead of MIT or Apache?

AGPLv3 is a strong copyleft license: derivative works must also be AGPLv3, and the "A" extends that requirement to users who interact with a modified version **over a network**.

The choice is deliberate, not ideological:

- **Protect derivative parametric models from closed-source forks.** Parametric model classes (gears, joints, brackets, mounts) are the long-tail value of this library. A permissive license would let a vendor fork the library, refine the models for a specific product line, and never give the improvements back. Copyleft means improvements flow back to the commons.
- **Close the SaaS loophole.** A plain GPLv3 would let someone wrap `vibe-cading` in a web service, modify it internally, and never publish the source — because the user never receives a copy of the binary. AGPLv3 section 13 closes that gap: if you let users interact with a modified version over a network, you must offer them the corresponding source.
- **Friendly to end-users, demanding of redistributors.** Hobbyists, makers, and engineers who use the library to design parts they print or sell are entirely unaffected by copyleft (see below). The license only "bites" when you redistribute modified source code or run it as a service.

If you need a permissively-licensed CAD library, [CadQuery](https://github.com/CadQuery/cadquery) itself is Apache-2.0.

## I just want to print parts. Does AGPLv3 affect me?

**No.** Printing physical parts from STEP / STL files generated by `vibe-cading` is not "conveying" the software in the copyright sense. You can:

- Run `vibe-cading` to generate STEP / STL files, freely.
- 3D-print, machine, or cast those parts.
- **Sell the printed parts.** The license does not restrict commercial use.
- Share the generated STEP / STL files. The output of a program is, in general, not a derivative work of that program — similar to how a document written in a GPL-licensed text editor is not a derivative work of the editor.

What you cannot do without complying with AGPLv3 is redistribute the **library source code** (or a modified version of it) — but if you're only printing parts, that doesn't apply.

## I modified a model class (or wrote a new one on top of `vibe-cading`). Now what?

If you **keep your changes to yourself** (private fork on your laptop, internal use inside your company), you have no obligation to publish anything. AGPLv3 is triggered by distribution / network deployment, not by use.

If you **distribute** the modified or extended library — pushing to a public GitHub fork, shipping it inside a product, including it in a downloadable bundle — then:

- Your modifications must also be released under AGPLv3.
- You must make the **complete corresponding source** available to recipients (typically by linking to a public repository).
- You must preserve the copyright notice and AGPLv3 header on the original files, and add your own attribution to files you change substantially.
- New `.py` files you add under `vibe_cading/`, `parts/`, or `tools/` must carry the AGPLv3 header (see [CONTRIBUTING.md §1](CONTRIBUTING.md#-1-license--cla)).

This applies whether your new class lives in a downstream fork or is shipped as a third-party plugin / package that imports from `vibe_cading`.

## I want to expose `vibe-cading` as a network service. What does AGPLv3 require?

This is where the "A" matters. If users interact with `vibe-cading` (or any modified version of it) **remotely over a network** — for example, a web app that lets users parameterize and download CAD models, or a hosted SaaS that builds parts on demand — AGPLv3 section 13 requires you to:

- Offer those remote users **a way to obtain the corresponding source code** of the version they're interacting with, including any modifications you made.
- The offer must be prominent and the source must be the actual deployed version, not a stale snapshot.

The cleanest way to comply is to publish your modified source on a public repository and link to it from the service's UI. If you ship the unmodified library as part of a larger network service, you still need to make the unmodified `vibe-cading` source available; in practice, a link to this repository at the commit you deployed is sufficient.

If your network service uses `vibe-cading` **only on the backend** and your users never interact with it (e.g. you generate a static STEP file as part of an internal build pipeline and ship the file to a customer), this is closer to the "I just want to print parts" case — the file is your output, not a network-exposed version of the library.

## How does the CLA relate to the license?

The two are complementary, not redundant.

- **AGPLv3** governs how *users and downstream redistributors* may use, modify, and redistribute the library.
- The **Contributor License Agreement** (CLA) governs how *upstream contributions* enter this repository. Every PR is checked by the [CLA Assistant bot](.github/workflows/cla.yml); first-time contributors sign with the magic phrase documented in [CONTRIBUTING.md §1](CONTRIBUTING.md#-1-license--cla). Signatures are recorded in [`signatures/version1/cla.json`](signatures/version1/cla.json).

By signing the CLA you confirm two things: (1) your contribution is licensed under AGPLv3, and (2) you have the right to submit it (i.e. you wrote it, or your employer has authorized you to contribute it). The CLA does **not** assign copyright to the project — you retain copyright on your contributions; the CLA only grants the project the licensing terms it needs to ship them under AGPLv3.

## Common confusions ("am I allowed to…?")

| Scenario | Allowed without AGPLv3 obligation? |
|---|---|
| Use `vibe-cading` to design a part and sell the printed part. | ✅ Yes. Output is yours; printing is not "conveying" the software. |
| Use `vibe-cading` to design a part and sell the STEP / STL file. | ✅ Yes. Same reasoning — the file is output, not a derivative of the library. |
| Fork `vibe-cading` on GitHub, add a custom model class, keep the fork public. | ⚠️ Fork must remain AGPLv3 and the new class must carry the AGPLv3 header. |
| Add `vibe-cading` as a dependency in a closed-source desktop CAD app you sell. | ❌ Not compatible with AGPLv3. The combined work must be AGPLv3. |
| Embed `vibe-cading` in a closed-source SaaS that lets users parameterize parts. | ❌ AGPLv3 section 13 requires you to offer source to remote users. |
| Use `vibe-cading` inside your company on an internal tool that never leaves the network. | ✅ Internal use is fine; no distribution = no source-distribution trigger. |
| Write a tutorial blog post with code snippets from `vibe-cading`. | ✅ Yes (fair use / quotation). Cite the project and link to the license. |
| Run `vibe-cading` in a CI job to generate STEP files that get shipped to customers as build artifacts. | ✅ Yes — the STEP files are output, the CI job is internal use. |

When in doubt, the question to ask is: **am I distributing the library code (or a modified version), or am I distributing output the library generated?** Output is yours; library code is AGPLv3.
